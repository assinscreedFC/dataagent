<objective>
Durcir le `sql_tool_node` existant (livré minimal en Phase 1) pour qu'il ne casse plus sur du SQL halluciné. Deux ajouts : (1) validation de la query générée contre le catalogue DuckDB via `EXPLAIN` AVANT de l'exécuter sur les données (D-01, D-02) ; (2) boucle de retry bornée par `SQL_MAX_RETRIES` qui re-prompte le LLM avec l'erreur DuckDB exacte pour régénérer une query corrigée (D-03, D-04). Couvre TOOL-01.

Purpose: Empêcher le crash silencieux ou les findings vides quand le LLM génère une query référençant une table/colonne inexistante. Le tool s'auto-corrige et, en dernier recours, produit un finding d'erreur honnête exploitable par le synthesizer.

Output: `sql_tool_node` durci (validation + retry), constante `SQL_MAX_RETRIES`, tests couvrant les 4 critères de succès.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM → DuckDB | Le SQL est généré par le LLM (à partir du schema réel) puis exécuté sur la connexion DuckDB. Surface principale de cette phase. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | `_execute_subquestion` (SQL LLM exécuté sur DuckDB) | mitigate | Validation `EXPLAIN` avant exec (parse+résout sans scanner) ; EXPLAIN ne mute pas les données. DB chargée depuis CSV locaux read-only, pas de DDL/DML dans les prompts (questions analytiques SELECT). |
| T-02-02 | Denial of Service | Boucle de retry intra-tool | mitigate | Retry borné par `SQL_MAX_RETRIES` (défaut 2) — pas de boucle infinie, hard stop garanti (D-03). |
| T-02-03 | Information Disclosure | Logs sur erreur SQL/validation | accept | Logge subquestion + sql + message d'erreur DuckDB ; pas de PII (dataset Olist synthétique en test, public en prod). Clé API jamais loggée. |
| T-02-04 | Injection | SQL généré par LLM | accept | Pas d'input utilisateur concaténé directement dans le SQL — la question passe par le LLM qui produit du SELECT sur schema connu. EXPLAIN valide les références au catalogue. Risque résiduel faible (DB locale, pas de secrets, pas de write). |
</threat_model>
